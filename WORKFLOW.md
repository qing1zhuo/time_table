# 项目工作流文档

本文档旨在为参与“周计划模板”项目的开发者提供清晰的工作流程和协作规范。

## 1. 环境搭建

在开始开发之前，请确保您的开发环境已正确配置。

### 1.1. 克隆仓库

首先，将项目仓库克隆到您的本地机器：

```bash
git clone <repository-url>
cd time_table
```

### 1.2. 创建虚拟环境

为了保持依赖隔离，我们强烈建议使用 Python 虚拟环境。

```bash
# 创建虚拟环境
python -m venv .venv

# 激活虚拟环境 (Windows)
.venv\Scripts\activate
```

### 1.3. 安装依赖

激活虚拟环境后，使用 `pip` 安装项目所需的所有依赖：

```bash
pip install -r requirements.txt
```

## 2. 开发流程

在进行代码修改或功能开发时，请遵循以下流程。

### 2.1. 运行开发模式

在开发过程中，您需要能够看到控制台输出的调试信息和错误日志。因此，请直接运行 `main.pyw` 文件，而不是使用 `run.vbs`。

```bash
# 确保虚拟环境已激活
python main.pyw
```

**注意**：在开发时，建议将 `main.pyw` 临时重命名为 `main.py`，这样可以更方便地在命令行中运行和调试。

### 2.2. 代码风格

-   请遵循 [PEP 8](https://www.python.org/dev/peps/pep-0008/) 代码风格指南。
-   保持代码整洁、可读，并添加必要的注释。

### 2.3. 分支管理

-   所有新功能开发和 Bug 修复都应在新的 `feature/` 或 `fix/` 分支上进行。
-   请勿直接向 `main` 分支提交代码。

```bash
# 创建并切换到新分支
git checkout -b feature/your-new-feature
```

### 2.4. 提交代码

-   在提交代码前，请确保所有测试都已通过。
-   使用清晰、有意义的提交信息。

```bash
git add .
git commit -m "feat: 添加了XX功能"
```

### 2.5. 发起合并请求

-   将您的分支推送到远程仓库，并发起一个合并请求 (Pull Request) 到 `main` 分支。
-   在合并请求中，请详细描述您所做的更改。

## 3. 架构概览

本项目采用混合桌面应用架构，核心组件包括：

-   **GUI**: 使用 `tkinter` 构建，位于 `main.pyw`。
-   **数据库逻辑**: 使用 `SQLite`，相关操作位于 `backend/db.py`。
-   **原生通知**: 使用 `plyer` 库实现。
-   **启动脚本**: `run.vbs` 和 `run_desktop_app.bat` 用于优化用户启动体验。

在进行开发时，请充分理解各组件的职责，以保持架构的一致性。

## 4. 注意事项

-   由于 `main.pyw` 是一个 GUI 应用，直接在命令行中运行可能不会立即返回。请确保您的开发工具和调试器能够正确处理这种情况。
-   在发布新版本时，请记得检查并更新 `README.md` 和 `WORKFLOW.md` 文档。
